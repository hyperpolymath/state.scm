== Executive Summary

This document outlines the path to formal standardization of the STATE
(Stateful Context Tracking Engine) format through appropriate standards
bodies. The recommended approach is a phased strategy starting with
community specification and progressing to formal standards registration.

== Standards Body Analysis

=== Evaluation Criteria

[cols="1,3"]
|===
|Criterion |Description

|Scope Fit
|How well the body's mandate matches STATE's purpose

|Process Complexity
|Time and effort required for standardization

|Recognition
|Industry recognition and adoption weight

|Precedent
|Similar specifications successfully standardized
|===

=== IETF (Internet Engineering Task Force)

[cols="1,4"]
|===
|Aspect |Assessment

|**Scope Fit**
|⭐⭐⭐⭐ Excellent - IETF handles data format specifications

|**Relevance**
|Strong precedent for interchange formats (JSON, iCalendar, GeoJSON)

|**Process**
|Internet-Draft → Working Group → RFC

|**Timeline**
|12-36 months typical

|**Track**
|Informational RFC (specification) or Standards Track (if protocol)
|===

**Recommended Path**: Informational RFC

**Rationale**: STATE is primarily a data format, not a protocol. Informational
RFCs are appropriate for specifications that document formats intended for
broad interoperability.

**Precedents**:

* RFC 8259 - The JSON Data Interchange Format
* RFC 5545 - iCalendar
* RFC 7946 - GeoJSON
* RFC 9110 - HTTP Semantics

[cols="1,4"]
|===

== Recommended Standardization Path

=== Phase 1: Community Specification (Current → +6 months)

**Objective**: Establish authoritative specification with community review

**Deliverables**:

1. ✅ ABNF Grammar (`spec/abnf/state.abnf`)
2. ✅ Format Specification (`spec/STATE-FORMAT-SPEC.adoc`)
3. ✅ JSON Schema (`spec/schema/state.schema.json`)
4. ✅ Version Policy (`spec/VERSION-POLICY.adoc`)
5. ⬜ Reference implementations in multiple languages
6. ⬜ Comprehensive test suite
7. ⬜ Interoperability testing results

**Actions**:

* [ ] Publish specification on GitHub with CC-BY-4.0 license
* [ ] Announce to AI/LLM developer communities
* [ ] Solicit feedback and incorporate changes
* [ ] Develop reference implementations (Rust, TypeScript, Python)
* [ ] Create conformance test suite

=== Phase 2: IANA Registration (+6 → +12 months)

**Objective**: Register media type for format identification

**Deliverables**:

1. Media Type Registration Template
2. Security Considerations Document
3. Fragment Identifier Specification

**Actions**:

* [ ] Prepare IANA Media Type registration
* [ ] Submit for Expert Review (provisional)
* [ ] Document in IETF Internet-Draft

**Registration Request**:

[source]
----
Type name: application
Subtype name: state+scheme
Required parameters: None
Optional parameters:
  charset: The character set. Default is UTF-8.
  version: Format version (e.g., "2.0")
Encoding: 8bit
Security: See specification Section 8
Interoperability: See specification Section 7
Published specification: [URL to spec]
Applications: AI assistants, project management tools
Fragment identifier: None defined
Additional info:
  Deprecated alias: None
  Magic number: "(define state"
  File extension: .scm, .state
Person to contact: [TBD]
Intended usage: COMMON
Restrictions: None
Author: Jonathan D.A. Jewell
Change controller: STATE Format Working Group
----

=== Phase 3: IETF Internet-Draft (+12 → +18 months)

**Objective**: Submit format specification to IETF

**Deliverables**:

1. Internet-Draft in RFC XML format
2. Implementation report (2+ implementations)
3. Interoperability report

**Actions**:

* [ ] Convert specification to RFC XML/v3 format
* [ ] Submit as Individual Internet-Draft
* [ ] Present to relevant Area Directors
* [ ] Seek Working Group adoption or pursue Independent Submission

**Potential Working Groups**:

* **DISPATCH** - For initial discussion and routing
* **SEDATE** - If JSON binding becomes primary
* **New WG** - If sufficient interest in AI interchange formats

=== Phase 4: RFC Publication (+18 → +30 months)

**Objective**: Publish as IETF RFC

**Deliverables**:

1. Approved RFC
2. IANA permanent media type registration
3. Implementation recommendations

**Track Options**:

[cols="1,3,2"]
|===
|Track |Description |Requirements

|Informational
|Documents specification for community use
|IESG review only

|Best Current Practice
|Recommends implementation approach
|IETF consensus

|Standards Track
|Full protocol/format standard
|Working Group, implementations
|===

**Recommended**: Informational RFC (lowest barrier, appropriate for format spec)

=== Phase 5: Ecosystem Integration (+30 → +48 months)

**Objective**: Broader ecosystem standardization

**Options**:

1. **W3C JSON-LD Context** - Semantic web integration
2. **ISO/IEC Submission** - Via national standards body
3. **Industry Consortium** - AI workflow interoperability group

== Alternative Paths

=== Path A: IETF-First (Recommended)

----
Community Spec → IANA Provisional → Internet-Draft → RFC → IANA Permanent
----

**Pros**: Well-understood process, strong precedent, broad recognition
**Cons**: Can be slow, requires IETF engagement

=== Path B: De Facto Standard

----
Community Spec → Reference Implementation → Adoption → Retroactive Standards
----

**Pros**: Fastest path to adoption, market-driven
**Cons**: No official recognition, fragmentation risk

=== Path C: Industry Consortium

----
Community Spec → Form Working Group → Consortium Standard → ISO Submission
----

**Pros**: Industry-driven, practical focus
**Cons**: Requires coalition building, governance overhead

== Appendix B: Document Formats

=== IETF Internet-Draft Format

Internet-Drafts must follow RFC 7991 (xml2rfc v3) or legacy format:

[source,xml]
----
<?xml version="1.0" encoding="UTF-8"?>
<rfc xmlns:xi="http://www.w3.org/2001/XInclude"
     docName="draft-jewell-state-format-00"
     category="info"
     ipr="trust200902">
  <front>
    <title>STATE Format Specification</title>
    <author fullname="Jonathan D.A. Jewell">
      <address>
        <email>email@example.com</email>
      </address>
    </author>
    <date year="2025" month="December"/>
    <abstract>
      <t>Specification for STATE files...</t>
    </abstract>
  </front>
  <middle>
    <!-- Specification content -->
  </middle>
  <back>
    <references>
      <!-- References -->
    </references>
  </back>
</rfc>
----
